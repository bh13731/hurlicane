# Plan: Fix Anthropic RateLimitError (429) in Deep Dives

## Problem Analysis

Sentry issue #7383088338: `RateLimitError` (429) from Anthropic API when generating deep dives with Claude Opus 4.6. The org rate limit is **30,000 input tokens per minute** for claude-opus-4-6.

**Root cause**: `generate_vc_deep_dives` in `dashboard_services/deep_dives.py` uses a `ThreadPoolExecutor` with `ANTHROPIC_WORKERS = 2` concurrent workers (from `lib/constants.py:46`). Each deep dive prompt includes firm context, 5 recent emails, summaries, Granola notes, and DD questions — easily 5,000–15,000 input tokens per call. Two concurrent requests can exceed the 30k tokens/minute org limit.

The existing `_with_anthropic_backoff` in `lib/anthropic_client.py` retries 3 times with delays of ~30s, ~60s, ~120s. However:
1. With 2 concurrent workers, both hit the rate limit simultaneously and both retry — doubling the request rate during recovery
2. Retries don't coordinate: worker A's retry can land at the same time as worker B's retry
3. 3 retries may be insufficient when the rate window hasn't fully replenished

**Evidence from breadcrumbs**: The 429 cascade at 21:11:53 → 21:12:28 → 21:13:32 shows the backoff delays were respected but the rate limit was still exceeded after all retries.

## Fix Strategy

### Step 1: Serialize Opus calls with a global threading semaphore
**File**: `lib/anthropic_client.py`

Add a module-level `threading.Semaphore(1)` that serializes all Opus-model calls. This ensures only one Opus request is in-flight at a time, preventing concurrent workers from competing for the same rate limit budget. Non-Opus models (Sonnet, Haiku) have higher rate limits and don't need this.

Wrap the API call in `claude_call` (and `claude_call_streaming`) with this semaphore when the model is `CLAUDE_OPUS`.

### Step 2: Increase retry budget for rate limits
**File**: `lib/anthropic_client.py`

Increase `max_retries` from 3 to 5 in `_with_anthropic_backoff` and increase `base_delay` from 30s to 60s. With the 30k tokens/minute limit, we need longer waits to let the rate window fully replenish. This gives delays of ~60s, ~120s, ~240s, ~480s, ~960s — ample time for even a completely exhausted budget.

### Step 3: Parse Retry-After header from 429 responses
**File**: `lib/anthropic_client.py`

When a `RateLimitError` is caught, check for `retry-after` header in the response. If present, use `max(header_value, computed_backoff)` as the delay. This respects the server's suggestion while maintaining our minimum backoff floor.

### Step 4: Add rate-limit skip tracking to deep dives
**File**: `dashboard_services/deep_dives.py`

When a firm analysis fails due to rate limit exhaustion, track it in the result as `rate_limited: True` (already partially done — the error handler at line 258-271 catches rate limits and skips Sentry). Add a count of rate-limited firms to the final result dict for observability.
