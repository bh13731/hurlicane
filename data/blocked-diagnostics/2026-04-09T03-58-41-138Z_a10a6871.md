# Plan

## Goal
Fix the WorkflowBlocked Sentry error by ensuring stale singleton recovery notes (`model-fallback`, `alt-provider`) are only treated as "in flight" when a matching recovery phase job for the same workflow/phase/cycle is actually active; otherwise the workflow must take a deterministic recovery outcome instead of remaining `running` until the gap detector blocks it generically.

## Assessment

**Refined Root Cause**: The risky paths are the singleton recovery keys in `src/server/orchestrator/WorkflowManager.ts` for `model-fallback` and `alt-provider`, not the normal `cli-retry` counter path.

The failure mode is:
1. A phase job fails and spawns a recovery attempt, which writes a singleton note such as `workflow/<id>/recovery/<phase>/cycle-<n>/model-fallback` or `.../alt-provider`.
2. Later, the same failed phase is reprocessed with `force: true` after that recovery attempt is already terminal, or after the note is left behind from an earlier recovery in the same phase/cycle.
3. `_onJobCompleted()` sees the existing singleton key and treats it as "recovery already in flight" even though no matching recovery job for that phase/cycle is still active.
4. The workflow remains `running` with no useful next action.
5. `reconcileRunningWorkflows()` eventually replays the failed phase job and blocks the workflow with `Workflow stuck after failed ...`, which is what Sentry reports.

**Review Improvements Applied**:
- The plan now requires the active-recovery check to match the same workflow, phase, and cycle instead of any active job on the workflow.
- The stale-note behavior is now explicit and testable. Because `model-fallback` and `alt-provider` use singleton keys, "continue normal handling" is too vague unless the implementation first clears/replaces the stale singleton or deliberately blocks with a specific exhaustion reason.
- Regression coverage now requires exact assertions on workflow status, blocked reason, and spawned-job count, not just the absence of duplicate jobs.

- `src/server/orchestrator/WorkflowManager.ts` — failed-job recovery logic around the singleton recovery keys and `reconcileRunningWorkflows()`
